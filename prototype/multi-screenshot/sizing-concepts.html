Image Sizing Concepts for Filmstrip Grid

How should screenshots of different sizes fit inside the composite?

Users capture screenshots of varying dimensions — a full-screen dashboard (2560×1440), a narrow mobile panel (390×844), a small tooltip crop (280×120). The filmstrip must handle all of these in the same grid. Below are five concepts showing how.

Concept 1 Fixed Frame — Scale to Fit (contain)

Every cell has the same fixed aspect ratio (16:10). The screenshot scales down to fit entirely inside the frame, preserving its aspect ratio. Mismatched images get letterbox bars (dark padding) on the sides or top/bottom.

Verdict: Workable but lossy. The grid is perfectly predictable — every cell is the same size, easy to lay out. But tall screenshots (like a mobile view) shrink dramatically to fit the wide frame, wasting space. The letterbox bars are visual noise that competes with annotations. And the screenshots aren't 1:1 — a user comparing pixel-level details loses fidelity. Works well when all screenshots happen to be similar aspect ratios. Falls apart with mixed orientations.
Concept 2 Natural Size — Every Image at Captured Dimensions

Each image renders at its exact captured pixel dimensions (or a uniform scale factor). No grid — images flow freely, each one exactly the size the user drew the selection rectangle. This is the "infinite canvas" approach taken literally.

Verdict: Faithful but chaotic. Perfect pixel fidelity — what you captured is exactly what you see. But the layout is unpredictable and messy. A tiny tooltip crop sits next to a massive dashboard capture. The stitched export would be enormous and irregular. Comparisons between images are hard because they're different physical sizes. The user has to scroll and hunt. This works for a freeform whiteboard tool, but not for a structured comparison filmstrip.
Concept 3 Uniform Row Height — Each Row Normalizes to the Same Height

All images in a row are scaled to the same height (the tallest image in that row sets the height). Each image keeps its native aspect ratio, so widths vary within the row. The grid columns flex to accommodate different widths. This is similar to how Google Photos or Flickr lay out image grids.

Verdict: Strong balance. Images keep their aspect ratios — no cropping, no letterboxing. Rows are visually tidy because everything shares a baseline height. Wide screenshots get more horizontal space naturally. Narrow screenshots don't shrink to nothing. The stitched export has clean row breaks. The tradeoff: column widths aren't uniform, so the grid feels less "structured" than Concept 1. But for comparing screenshots, preserving the actual proportions matters more than grid neatness.
Concept 4 Masonry — Pinterest-Style Stacking

Fixed column widths (3 columns). Each image scales to fill the column width, keeping its aspect ratio, and stacks vertically with no row alignment. Tall images take more vertical space. Short images leave room for the next one. Classic masonry layout.

Verdict: Pretty but wrong for this use case. Masonry looks great for browsing photos — but Vibeliner is a comparison and annotation tool, not a gallery. The staggered heights make it hard to compare two images side-by-side because they don't share horizontal baselines. Annotation badges that reference "compare Image 1 to Image 3" are hard to follow when they're at completely different vertical positions. The reading order (column-first instead of row-first) is unintuitive for a numbered annotation sequence. This layout fights the core workflow.
Concept 5 Equal Columns + Aspect Fit — Grid Structure with Natural Proportions

Fixed equal-width columns (like Concept 1), but each image keeps its own aspect ratio within the column. The row height is set by the tallest image in that row — shorter images are vertically centered in the row with subtle background fill behind them. No cropping. No letterboxing. The grid stays structured but respects each screenshot's shape.

Verdict: Recommended. This is the best of both worlds. Equal column widths give a clean, predictable grid structure (easy to scan, easy to annotate across). Aspect ratio preservation means no information is lost — the user sees exactly what they captured. Row height flexes to the tallest image, so tall screenshots don't get crushed. Shorter images center cleanly in the extra space. The stitched export has consistent column widths, making it easy for LLMs to parse. The only cost: some vertical whitespace when images in the same row have very different heights. That's a good tradeoff — whitespace is better than cropping or distortion.
